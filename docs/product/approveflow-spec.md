# ApproveFlow — спецификация продукта

Краткий контекст продукта: `approveflow-project-overview.md`. Здесь — правила, сущности и переходы задачи; цепочка в overview упрощена, детальная машина состояний — ниже.

## Суть продукта

Одно **workspace** для агентства и приглашённого клиента: **задачи**, **этапы**, **согласования**, **история**, **комментарии**, понятно **кто сейчас блокирует** и **что происходит с проектом в целом** (агрегация задач и лента событий).

**Ядро** — жизненный цикл **одной задачи** и таблица переходов (раздел «Матрица переходов»). Проект, сдвиг дедлайнов и «вклад» — **надстройки**, не меняющие эту таблицу.

## Принципы

- В одном workspace у пользователя **ровно одна** роль; **admin** — полный доступ в workspace и те же переходы статусов задачи, что у **manager**, где не сказано иначе.
- Статусы задачи на стороне агентства меняют только **admin** и **manager**; **member** статусы **не** меняет (работа + комментарии; воронку ведёт manager; если человек и ведёт задачи по статусам — в системе это **manager**).
- **Двустороннее согласование у границы с клиентом:** без пары явных действий нет перехода в полноценное клиентское ревью и нет финального `done` (см. `client_handoff` и `pending_closure`).
- **Клиент:** видит задачи приглашённых проектов на **всех** этапах; **комментарии** на любом этапе; **смена статуса** — только там, где в таблице указано **C**.

## Роли

| Роль (код) | Кто | Зона |
|------------|-----|------|
| admin | директор / владелец | workspace + переходы как у manager |
| manager | PM, account | воронка задач, клиент, финальное закрытие |
| member | исполнитель | работа, комментарии, без смены статусов |
| client_view | клиент | чтение, комментарии; действия **C** по матрице переходов |

## Объекты и сущности

**Базовый слой**

- Workspace, WorkspaceMember (одна роль на пользователя в workspace).
- Project — контейнер работ; к проекту привязаны задачи.
- Task — статус, исполнитель (опционально), срок **due** (опционально), описание.
- Comment.
- TaskEvent — каждая смена статуса (кто, когда, from → to).
- **ClientHandoffAck** — клиент подтвердил приём к согласованию: `client_handoff` → `client_approval`.
- **ClientApprovalAction** — из `client_approval`: `approved` → `pending_closure` или `changes_requested` → `production`; **обязательный комментарий** только при `changes_requested`.
- **Подтверждение закрытия** от A/M — `pending_closure` → `done` (TaskEvent или отдельная запись).

Этап **brief** без отдельного документа «Brief» — это статус задачи + описание + комментарии.

**Следующий слой** (не меняет матрицу переходов)

- **История смены due:** кто, старое/новое значение, опционально комментарий. Клиент **не меняет due** напрямую: запрос через комментарий → решение **A/M**.
- **Лента активности проекта:** TaskEvent + комментарии + смены due.

## Статусы задачи

1. `brief`  
2. `production`  
3. `internal_review`  
4. `client_handoff` — отправили клиенту; ждём явного подтверждения клиента.  
5. `client_approval` — клиент: согласовать или запросить правки.  
6. `pending_closure` — клиент согласовал; ждём подтверждения закрытия от **A/M**.  
7. `done`

Из `done` никуда не переходим в первой версии продукта.

## Матрица переходов

**A** = admin, **M** = manager, **C** = client_view. **Member** переходов не выполняет.

| Из | В | Кто | Смысл |
|----|---|-----|--------|
| brief | production | A, M | старт работы |
| production | internal_review | A, M | к внутренней проверке |
| internal_review | production | A, M | внутренние правки |
| internal_review | client_handoff | A, M | агентство отправляет на согласование клиенту (только выбранные клиенты проекта) |
| client_handoff | internal_review | A, M | отозвать до повторной отправки |
| client_handoff | client_approval | C | клиент подтвердил приём к согласованию; ClientHandoffAck |
| client_approval | pending_closure | C | клиент согласовал (approve); закрытие ещё не финально |
| client_approval | production | C | Request changes + обязательный комментарий; ClientApprovalAction `changes_requested` |
| pending_closure | done | A, M | подтверждение закрытия менеджером/админом |

**Запрещено:** пропускать статусы; `internal_review` → `client_approval` напрямую; `client_approval` → `done` напрямую; любые переходы вне таблицы; выход из `done`.

**Двусторонние мосты**

- В `client_approval` только после **A/M:** `internal_review` → `client_handoff` и **C:** `client_handoff` → `client_approval`.
- В `done` только после **C:** approve → `pending_closure` и **A/M:** `pending_closure` → `done`.

## Действия в UI (имена кнопок)

- **`client_handoff` (клиент):** «Принять к согласованию» → `client_approval`.
- **`client_approval` (клиент):** «Согласовать» → `pending_closure`; «Запросить правки» → `production` + комментарий.
- **`pending_closure` (manager/admin):** «Подтвердить закрытие» → `done`.
- **A/M:** остальные переходы из матрицы.

## Видимость и комментарии

- **client_view:** чтение задачи на всех статусах; комментарии на любом этапе; смена статуса — только по строкам с **C** в матрице.
- **member:** без смены статусов; минимум — назначенные задачи и проекты, куда пользователь добавлен.

## Проект: статус и итерации

- Статус проекта можно не моделировать отдельной машиной: **агрегаты по задачам** — например, сколько задач в `client_handoff`, в `client_approval`, не в `done`, с просроченным `due`.
- Решение «идём дальше или нет» на уровне проекта сначала — комментарии + новые или открытые задачи.
- Статус проекта (`active` / `paused` / `completed`) в MVP — **только информационный**: не блокирует создание задач, смену статусов, комментарии и смену due. Блокировки определяются только матрицей переходов задачи.
- Итерации: **`internal_review` ↔ `production`** и **`client_approval` → `production`** при правках.

## Дедлайны и компромисс

- **due** меняют только **A/M**.
- Клиент просит сдвиг **комментарием**; **A/M** выставляет новый due; запись в **историю due**.

## «Живость»: вклад и кто блокирует

- Роль в workspace — WorkspaceMember; в задаче — **исполнитель**, **кто менял статус** (TaskEvent), **комментарии**.
- **Кто блокирует смену статуса:** например `client_handoff` → ждём действия **C**; `pending_closure` → ждём **A/M**; `internal_review` (чтобы уйти в `client_handoff` или в `production`) → ждём **A/M**. (Работа у исполнителя параллельно этому не описывается как «блокировка статуса».)
- Тяжёлая аналитика вклада и SLA — позже, см. overview.

## Сценарии приёмки

**A** — Manager создаёт project и task в `brief`, назначает member.

**B** — Manager: `brief` → `production` → … → `internal_review`; при необходимости `internal_review` ↔ `production`; затем `internal_review` → `client_handoff`.

**C** — Клиент: `client_handoff` → `client_approval`; затем либо Request changes → `production`, либо Approve → `pending_closure`.

**D** — Manager/admin: `pending_closure` → `done`. Цепочка в TaskEvent и записях согласий.

## Порядок реализации

1. БД и сущности базового слоя.  
2. **assertTransition(role, from, to)** по матрице + TaskEvent.  
3. API задач, проектов, комментариев.  
4. UI карточки задачи и кнопок.  
5. Приглашение **client_view** и проверки видимости.  
6. Экран проекта: агрегаты + лента (события, комментарии, история due).  
7. Уведомления, вложения, SLA/аналитика.

## Вне скоупа (пока)

- Кастомные воркфлоу на проект.  
- Периоды планирования (timeboxing задач по периодам).
- **Post-MVP epic:** SLA, эскалации, тяжёлая аналитика cycle time (мониторинг просрочек, авто-эскалация, отчёты по cycle time).  
- Вложения — сразу после ядра, если критично.  
- Сложный мультитенант-онбординг без упрощений.
